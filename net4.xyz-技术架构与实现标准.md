## 五、AFC 公链核心架构

### 5.1 PUE 共识机制（Proof of Useful Energy）

**核心创新**：将电力转化为对人类有用的 AI 智能，而非无意义的哈希计算。

#### POUE 工作流程

```
1. AFC 网络分配 AI 任务（知识推理/内容审核/科学计算）
   ↓
2. 节点的 AI 分身执行推理任务
   ↓
3. 提交结果哈希 + ZK-ML 证明（零知识机器学习）
   ↓
4. 其他节点验证（抽查机制）
   ↓
5. 验证通过 → 获得区块奖励 + 手续费分成
```

#### ZK-ML 验证
- 节点无需暴露原始数据和模型参数
- 通过零知识证明确认"AI 确实完成了任务"
- 防止节点作弊（伪造 AI 结果）

### 5.2 三级节点架构

| 节点类型 | 数量 | 准入条件 | 职责 | 收益分配 |
|---------|------|----------|------|----------|
| **核心验证节点** | 21 个 | - 购买 10 个火种（$10 万 USDT）<br>- 高性能 GPU（H100/A100）<br>- 32 核 CPU/256G 内存/1T SSD<br>- 社区投票排名前 21 | 出块、交易验证、链上治理、AI 任务分配 | 区块奖励（年化 8%-12%）+ 50% 手续费分成 |
| **子节点** | 128 个 | - 购买 1 个火种（$9,999 USDT）<br>- 消费级 GPU（RTX 4090）<br>- 稳定运行 ≥30 天 | 数据同步、交易广播、辅助验证、AI 任务执行 | 区块奖励（年化 3%-5%）+ 空投 |
| **普通节点** | 余下配额<br>（总上限 1 万） | - 持有 1 个火种 NFT（$9,999 USDT） | 存储轻节点数据、参与社区治理投票 | 生态空投（项目代币） |

### 5.3 AI 分身节点要求

| 节点类型 | AI 分身要求 | 模型选择 |
|---------|-------------|----------|
| 核心验证节点 | 高阶 AI 分身（复杂逻辑推理与决策） | Llama 3 70B/405B（本地部署） |
| 子节点 | 中阶 AI 分身（专项处理 NLP/CV） | Llama 3 70B 或 Mixtral 8x22B |
| 普通节点 | 轻量 AI 分身（基础交互） | GPT-4o/Claude 3.5（API 接入） |

**双轨制设计**：
- **轨道 A**：云端 API（轻节点/普通用户）- 低延迟、免维护
- **轨道 B**：本地部署（核心/子节点）- 数据主权、完全去中心化

## 十、合规与安全方案

### 10.1 支付合规

| 场景 | 流程设计 | 合规优势 |
|------|----------|----------|
| 初级会员（$99） | 信用卡支付 → Transak 网关 → 兑换 USDT → 转入 AFC 财务钱包 | 信用卡记录留存，便于税务申报 |
| 火种 NFT（$9,999） | 信用卡/银行转账 → 合规交易所 → 购买 USDT → 转入用户钱包 | 大额交易通过交易所 KYC，符合 AML 要求 |
| 书籍购买 | 信用卡/USDT → 作者钱包（自动分账，平台抽成 1%） | 链上可追溯，保护知识产权 |

**风控机制**：
- 信用卡通道仅限购买「封装版火种」（锁定 90 天不可转让）
- 超过 $5,000 必须使用加密货币支付
- 大额交易（≥$10 万）需 KYC 验证

### 10.2 证券属性规避

| 风险点 | 规避方案 |
|--------|----------|
| 承诺「年化收益/分红」 | 删除「理财/利息」字眼，改为「网络贡献奖励」或「Gas 费返还」 |
| NFT 被认定为证券 | 定义为「技术服务凭证」，而非投资标的 |
| 代币属性 | 强调购买的是 NFT（非证券），收益与节点贡献挂钩 |

### 10.3 安全防护

| 风险点 | 应对措施 |
|--------|----------|
| 智能合约漏洞 | 三重审计（CertiK + PeckShield + 社区赏金），测试网运行 ≥3 个月 |
| 支付合规风险 | 信用卡通道限制额度，超额必须 Crypto 支付 |
| AI 作恶风险 | 节点 AI 分身输出需通过内容安全过滤，违规扣减火种 |
| 私钥管理 | 核心节点私钥采用「冷存储 + 多签」，禁止热钱包存储大额资产 |
| DDoS 攻击 | Cloudflare 等防护服务 |
| 数据备份 | IPFS 多节点 Pinning，防止单点失效 |

### 10.4 数据隐私合规

- 欧盟用户支持 GDPR（数据可删除权）
- 存储数据加密（AES-256）
- 核心节点 KYC（Passport/ID 验证），大额交易上报 Chainalysis

## 十一、关键补充：查漏补缺

### 11.1 数据可用性（DA）层

| 项目 | 描述 |
|------|------|
| 现状 | 提到 IPFS/Arweave 存储，但未说明数据如何保证永久可读 |
| 方案 | 引入 Celestia 或 EigenDA 作为数据可用性层。AFC 主网只存状态根，原始数据扔给 DA 层，降低成本 |

### 11.2 ZK-ML 验证

| 项目 | 描述 |
|------|------|
| 现状 | PUE 机制要求验证 AI 确实工作了，如何防止节点作弊？ |
| 方案 | 采用 ZK-ML（零知识机器学习）。节点提交计算结果的同时，提交一个 ZK Proof，证明「这个结果确实是这个模型跑出来的」 |

### 11.3 跨链桥接

| 项目 | 描述 |
|------|------|
| 现状 | 火种 NFT 在 Base 上销售，需迁移至 AFC 主网 |
| 方案 | 开发双向锚定桥（Two-way Peg）。支持资产转移及 AFC 主网资产回退至 Base（以防主网故障） |

### 11.4 种子轮与流动性

| 项目 | 描述 |
|------|------|
| 现状 | 1 万火种卖完后，AFC 主网代币如何定价？ |
| 方案 | 预留 10%-15% 的代币作为流动性池（LP）注入 Uniswap/Base，确保火种 NFT 持有者能随时变现 |

| `MirromeSocial.sol` | - | 分布式社交内容存储与激励分发 |

### 17.2 PUE_Consensus.sol 核心逻辑（伪代码）

```solidity
// 核心函数：提交 AI 工作证明
function submitWorkProof(
    uint256 nodeId, 
    bytes32 taskHash, 
    bytes memory aiResult, 
    bytes memory zkProof // 零知识证明，证明 AI 运算真实发生
) external returns (bool) {
    // 1. 验证节点是否在活跃列表中
    require(NodeRegistry.isActive(nodeId), "Node not active");
    
    // 2. 验证任务哈希是否匹配当前区块高度分配的任务
    require(currentTasks[nodeId] == taskHash, "Task mismatch");
    
    // 3. 验证零知识证明（验证 AI 确实做了事，且没作弊）
    require(zkVerifier.verify(zkProof, aiResult), "Invalid ZK proof");
    
    // 4. 更新节点信誉值
    updateReputation(nodeId, +1);
    
    // 5. 铸造奖励
    _mint(AFC_Token, nodeId, rewardAmount);
    
    emit WorkSubmitted(nodeId, taskHash);
}
```

- **Data Format**：规定输入（Prompt/Data）和输出（Result Hash）的标准 JSON 格式
